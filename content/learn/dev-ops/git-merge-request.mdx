মার্জ রিকোয়েস্ট (Merge Request) বা Pull Request হলো গিট-ভিত্তিক ভার্সন কন্ট্রোল প্ল্যাটফর্মে (যেমন GitHub, GitLab, Bitbucket) কোড পরিবর্তনকে মূল ব্রাঞ্চে যুক্ত করার জন্য একটি আনুষ্ঠানিক অনুরোধ। এটি টিমওয়ার্ক, কোড রিভিউ, ও স্বচ্ছতা নিশ্চিত করে।

## কেন মার্জ রিকোয়েস্ট দরকার?
- **কোড রিভিউ:** অন্য ডেভেলপাররা আপনার কোড দেখে মন্তব্য/পরামর্শ দিতে পারে।
- **বাগ/সমস্যা ধরতে:** একাধিক চোখে কোড দেখলে ভুল ধরা সহজ হয়।
- **টিমওয়ার্ক:** একাধিক ডেভেলপার একসাথে কাজ করতে পারে, কিন্তু মূল কোডে পরিবর্তন আনতে হলে সবাইকে জানাতে হয়।
- **স্বচ্ছতা:** কে, কখন, কী পরিবর্তন করেছে—সবকিছু ট্র্যাক করা যায়।

## বাস্তব উদাহরণ: মার্জ রিকোয়েস্টের জীবনচক্র

ধরা যাক, আপনি একটি ফিচার ব্রাঞ্চে কাজ করছেন—`feature/login-page`।

1. **ব্রাঞ্চ তৈরি:**
   ```bash
   git checkout -b feature/login-page
   ```
2. **কোড পরিবর্তন ও কমিট:**
   ```bash
   # কোড এডিট করুন
   git add .
   git commit -m "Login page added"
   ```
3. **রিমোটে পুশ করুন:**
   ```bash
   git push origin feature/login-page
   ```
4. **মার্জ রিকোয়েস্ট তৈরি:**
   - GitHub/GitLab-এ গিয়ে `feature/login-page` থেকে `main` বা `develop` ব্রাঞ্চে মার্জ রিকোয়েস্ট দিন।
   - টাইটেল, বিবরণ লিখুন, রিভিউয়ার অ্যাসাইন করুন।

5. **কোড রিভিউ:**
   - টিমের অন্য সদস্যরা কোড দেখে মন্তব্য/পরামর্শ/পরিবর্তন চাইতে পারে।
   - Automated test, CI/CD pipeline চলতে পারে।

6. **পরিবর্তন প্রয়োজনে:**
   - রিভিউয়ারের পরামর্শে কোড পরিবর্তন করুন, আবার কমিট ও পুশ করুন।
   - মার্জ রিকোয়েস্টে নতুন কমিট যুক্ত হবে।

7. **মার্জ:**
   - সবকিছু ঠিক থাকলে, মার্জ রিকোয়েস্ট Approve হবে।
   - তারপর মূল ব্রাঞ্চে মার্জ করা হবে।

## মার্জ রিকোয়েস্টের সুবিধা
- **Collaborative Development:** একাধিক ডেভেলপার একসাথে কাজ করতে পারে, কিন্তু মূল কোডে পরিবর্তন আনতে হলে সবাইকে জানাতে হয়।
- **Code Quality:** কোড রিভিউয়ের মাধ্যমে কোডের মান নিশ্চিত হয়।
- **Issue Tracking:** কোন ফিচার/বাগ ফিক্স কখন, কে করেছে—সবকিছু ট্র্যাক করা যায়।
- **Automated Testing:** CI/CD pipeline মার্জ রিকোয়েস্টে স্বয়ংক্রিয়ভাবে টেস্ট চালাতে পারে।

## GitHub-এ মার্জ রিকোয়েস্টের ধাপ
1. **Fork/Clone করুন রিপোজিটরি।**
2. **নতুন ব্রাঞ্চে কাজ করুন।**
3. **কোড পরিবর্তন, কমিট, ও পুশ করুন।**
4. **GitHub-এ গিয়ে Pull Request (PR) তৈরি করুন।**
5. **রিভিউ, আলোচনা, পরিবর্তন, ও মার্জ।**

## টিপস
- মার্জ রিকোয়েস্টে সবসময় স্পষ্ট টাইটেল ও বিবরণ লিখুন।
- ছোট ছোট পরিবর্তন আলাদা ব্রাঞ্চে করুন, যাতে রিভিউ সহজ হয়।
- Automated test ও linting চালান, যাতে কোডের মান বজায় থাকে।

## বাস্তব অভিজ্ঞতা

> "আমাদের টিমে সবাই আলাদা ফিচার ব্রাঞ্চে কাজ করে। কাজ শেষ হলে মার্জ রিকোয়েস্ট দেয়। টিম লিড রিভিউ করে, Automated test চলে, তারপর Approve হলে মূল ব্রাঞ্চে মার্জ হয়। এতে কোডের মান, স্বচ্ছতা, ও টিমওয়ার্ক বজায় থাকে।"

## সংক্ষেপে
- মার্জ রিকোয়েস্ট টিমওয়ার্ক, কোড রিভিউ, ও স্বচ্ছতা নিশ্চিত করে।
- এটি ছাড়া বড় প্রজেক্টে কোড ম্যানেজ করা কঠিন।
- GitHub/GitLab-এ মার্জ রিকোয়েস্টের মাধ্যমে Collaborative Development সহজ হয়।
