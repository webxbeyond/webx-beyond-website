গিটে ব্রাঞ্চ মার্জ মানে হলো—একটি ব্রাঞ্চের পরিবর্তন অন্য ব্রাঞ্চে "মিশিয়ে" দেওয়া। সাধারণত ফিচার/বাগ ফিক্স ব্রাঞ্চের কাজ শেষ হলে, সেটি মূল ব্রাঞ্চে মার্জ করা হয়।

## কেন ব্রাঞ্চ মার্জ করবেন?
- ফিচার/বাগ ফিক্সের কাজ শেষ হলে, মূল কোডবেসে যুক্ত করতে।
- টিমওয়ার্কে সবাই আলাদা ব্রাঞ্চে কাজ করে, শেষে মার্জ করে।
- কোডের ইতিহাস ও পরিবর্তন সংরক্ষণ করতে।

## মার্জের ধাপ

ধরা যাক, আপনি `feature/login-page` ব্রাঞ্চে কাজ করেছেন, এখন সেটি `main`-এ মার্জ করতে চান।

### ১. মূল ব্রাঞ্চে সুইচ করুন
```bash
git checkout main
```

### ২. মার্জ করুন
```bash
git merge feature/login-page
```

- এতে `feature/login-page`-এর সব পরিবর্তন `main`-এ যুক্ত হবে।

### ৩. কনফ্লিক্ট হলে
- গিট জানিয়ে দেবে কোন ফাইল কনফ্লিক্ট হয়েছে।
- কনফ্লিক্ট ম্যানুয়ালি ঠিক করুন, তারপর:
  ```bash
  git add <conflicted-file>
  git commit
  ```

### ৪. রিমোটে পুশ করুন
```bash
git push origin main
```

## মার্জের ধরন
- **Fast-forward merge:** যদি মূল ব্রাঞ্চে কোনো নতুন কমিট না থাকে, সরাসরি ফিচার ব্রাঞ্চের কমিট যুক্ত হয়।
- **Three-way merge:** দুই ব্রাঞ্চে আলাদা কমিট থাকলে, নতুন মার্জ কমিট তৈরি হয়।

## বাস্তব উদাহরণ

> "আমরা টিমে সবাই আলাদা ফিচার ব্রাঞ্চে কাজ করি। কাজ শেষে টিম লিড মূল ব্রাঞ্চে মার্জ করে, যাতে কোডবেস আপডেট থাকে। কনফ্লিক্ট হলে সবাই মিলে সমাধান করি।"

## টিপস
- মার্জের আগে মূল ব্রাঞ্চ আপডেট (pull) করুন।
- কনফ্লিক্ট হলে, টিমে আলোচনা করে সমাধান করুন।
- মার্জ কমিটে স্পষ্ট ম্যাসেজ লিখুন।

## সতর্কতা
- মার্জ করার আগে সবকিছু ব্যাকআপ রাখুন।
- ভুল মার্জ হলে, reset/revert ব্যবহার করুন।

## সংক্ষেপে
- ব্রাঞ্চ মার্জ গিটে টিমওয়ার্ক, ফিচার ডেলিভারি, ও কোডবেস আপডেট রাখতে অপরিহার্য।
- Fast-forward ও Three-way—দুই ধরনের মার্জ আছে।
- কনফ্লিক্ট হলে, সতর্কভাবে সমাধান করুন।
