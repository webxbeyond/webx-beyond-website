## ভূমিকা

এই লেসনে আমরা পডম্যান এবং ডকার—দু’টি জনপ্রিয় কনটেইনার টুলের মধ্যে মিল এবং অমিল পর্যায়ক্রমে তুলে ধরব। উদ্দেশ্য হলো আপনি বুঝতে পারবেন কখন কোনটি বেছে নেবেন, কীভাবে পরিবেশ পরিবর্তন করলে সমস্যা হতে পারে, এবং সাধারণ ভিন্নতার পেছনের কারণগুলো।

এই পাঠটি নীচের বিষয়গুলো কভার করবে:

- পডম্যান ও ডকার কোথায় একই (সম্ভব আহত ব্যবহার)
- মূল আর্কিটেকচারিক পার্থক্য (daemon, rootless ইত্যাদি)
- নিরাপত্তা এবং অপারেশনাল পার্থক্য
- বাস্তব ডেভ/প্রোড মাইগ্রেশন বিবেচ্য বিষয়
- সহজ উপমা (analogy) যা জটিল ধারণাকে সরল করে
- একটি প্র্যাকটিক্যাল এক্সারসাইজ যা কনসেপ্টগুলো হাতে-কলমে যাচাই করবে

নোট: এই লেসনটি ধারণাগত এবং তুলনামূলক; ইনস্টলেশন-সংক্রান্ত কমান্ড ও হ্যান্ডস-অন ধাপে পরবর্তী লেসনে বিস্তারিতভাবে দেখানো হবে।

## সংক্ষেপে TL;DR

- মিল: উভয়ই OCI-কমপ্লায়েন্ট ইমেজ ব্যবহার করতে পারে এবং বহু CLI সাব-কমান্ড সাদৃশ্যপূর্ণ।
- পার্থক্য: পডম্যান daemon-less ও rootless অপশনকে কেন্দ্র করে ডিজাইন করা; ডকার সাধারণত centralized daemon-ভিত্তিক।
- সিদ্ধান্তের চাবিকাঠি: নিরাপত্তা ও লিনাক্স-মুখী সার্ভারে পডম্যান ভাল, টিম বা টুলিং যদি Docker-সেন্ট্রিক হয় তবে ডকার সহজ থাকতে পারে।

## মিল (Similarities)

1. OCI-কমপ্লায়েন্স: উভয়ই আধুনিক কনটেইনার ইকোসিস্টেমের স্ট্যান্ডার্ড অনুসরণ করে, ফলে ইমেজ একে অপরের সাথে বিনিময়যোগ্য।
2. CLI অভিজ্ঞতা: বেসিক কমান্ড (run, pull, push, images, ps, logs ইত্যাদি) অনেকটাই একরকম; তাই Docker ব্যবহারকারী পডম্যান দ্রুত শিখে ফেলতে পারে।
3. ইকো-সিস্টেম ইন্টারঅপারেবিলিটি: ইমেজ, রেজিস্ট্রি এবং অনেক টুল (CI/CD, কিউবেরনেটিস ইত্যাদি) উভয়ের সাপোর্ট করে।

এই মিলগুলো শেখে আপনাকে দ্রুত শুরু করতে সাহায্য করে—একই ইমেজকে বিভিন্ন টুল দিয়ে চালানো সম্ভব।

## বড় পার্থক্যগুলো (Differences)

নীচে প্রতিটি পার্থক্যকে সহজ ভাষায় এবং বাস্তব ফলাফলের দিক থেকে ব্যাখ্যা করা হলো।

1) Daemon vs Daemon-less
- Docker: একটি কেন্দ্রীয় background সার্ভিস (dockerd) থাকে যা কনটেইনার ম্যানেজ করে।
- Podman: daemon-less ডিজাইন—প্রতিটি কমান্ড আলাদা প্রক্রিয়া হিসেবে চলে।

প্রভাব: daemon থাকা মানে ক্লাস্টার-লেবেল shared state সহজ; কিন্তু daemon-less থাকলে সার্ভিস ফেইলারের ঝুঁকি কমে এবং অপারেশনাল সিকিউরিটি বাড়ে।

2) Rootful vs Rootless
- Docker: ঐতিহ্যগতভাবে ডকার daemon-কে রুট অধিকার দিয়ে চালানো হয়েছে (যদিও rootless Docker-ও আছে)।
- Podman: মূলত rootless অপশনকে গুরুত্ব দেয়—ব্যবহারকারী অধিকারেই কনটেইনার চালাতে পারে।

প্রভাব: rootless থাকলে হোস্ট সিস্টেমে ভুল করে উচ্চাধিকারী পরিবর্তন হওয়ার সম্ভাবনা কমে। কিন্তু কিছু নেটওয়ার্ক/স্টোরেজ কনফিগারেশনে অতিরিক্ত সেটআপ লাগতে পারে।

3) Compatibility / CLI
- বেসিক CLI অনেকটাই মিল, কিন্তু advanced behaviour (systemd integration, generate unit, pods) এ পডম্যানের কিছু আলাদা কমান্ড ও সাবটুল আছে।

প্রভাব: সাধারণ কাজ দ্রুত রাখতে পারবেন; কিন্তু automation বা স্ক্রিপ্টে subtleties থাকতে পারে।

4) Architecture & Integration
- Docker Engine এক জায়গায় centralized; Podman `Buildah`, `Skopeo` ইত্যাদি টুলের সাথে কোর ভিত্তি ভাগ করে।

প্রভাব: টুলিং মডুলার; কিছু workflow-এ পডম্যান বেশি নমনীয় হতে পারে (উদাহরণ: বিল্ড স্টেপ আলাদা করা)।

5) Windows / macOS পার্থক্য
- Docker Desktop খুব পরিচিত এবং GUI সহ সহজ।
- Podman macOS/Windows-এ Podman Machine বা Podman Desktop দিয়ে চলে — অভিজ্ঞতা ভিন্ন হতে পারে।

প্রভাব: Cross-platform ডেভেলপমেন্টে টিমের পছন্দ ও সরঞ্জামের প্রয়োজন অনুযায়ী সিদ্ধান্ত নেবেন।

6) Resource ও পারফরম্যান্স
- দুইটির পারফরম্যান্স নির্ভর করে কনফিগারেশন ও হোস্ট অপারেটিং সিস্টেমের ওপর। কিছু ক্ষেত্রে daemon-less হওয়ায় নেটিভ ল্যাটেন্সি কমতে পারে; অন্য ক্ষেত্রেও centralized caching সুবিধা ডকারকে দ্রুত করতে পারে।

## সহজ উপমা (Analogy) — সমস্যা বোঝার জন্য

ধান্যতা: কল্পনা করুন—ডকার হলো একটি বড় ট্রেন-স্টেশন যেখানে সব পণ্য ট্রেনে ঢোকানো ও নামানো হয়; স্টেশন (daemon) সব কন্ট্রোল করে। পডম্যান হলো বহু ছোট ট্রাকের একটি বহর—প্রতি ট্রাকে চালক আছে এবং তারা স্বাধীনভাবে পণ্য নিয়ে চলে; কোনো একক স্টেশন সবকিছু নড়াচড়া করে না।

উপকারিতা:

- স্টেশন (Docker daemon) থাকলে logistics centralized হয়—সমন্বয় সহজ; কিন্তু যদি স্টেশন বন্ধ হয়ে যায়, সবকিছু থেমে যেতে পারে।
- ট্রাকগুলির (Podman) ক্ষেত্রে অনেক স্বাধীন চালক থাকলে একটি ট্রাকের সমস্যায় বাকি চালকরা কাজ চালিয়ে যেতে পারে—নেটওয়ার্কিং বা সিকিউরিটি কনফিগারেশনে স্বাধীনতা বাড়ে।

এই উপমা daemon এবং daemon-less আর্কিটেকচারের পার্থক্যকে সরল করে।

## মাইগ্রেশন ও বাস্তব বিবেচ্য বিষয়

যদি আপনি Docker থেকে Podman-এ বা উল্টো মাইগ্রেট করতে চান, নিম্নলিখিত জিনিসগুলো বিবেচনা করুন:

1. Automation স্ক্রিপ্ট: `docker`-কমান্ড ব্যবহার করে লেখা স্ক্রিপ্টগুলো সবসময় একইভাবে কাজ নাও করতে পারে—টেস্টিং জরুরি।
2. Systemd integration: Podman-এর `generate systemd` সুবিধা আছে; কিন্তু পুরো সিস্টেম-কনফিগারেশন যাচাই করা প্রয়োজন।
3. Rootless constraints: কিছু নেটওয়ার্কিং বা ভলিউম টাস্ক হলে অতিরিক্ত কনফিগারেশন লাগতে পারে (UID mapping, subuid/subgid)।
4. CI/CD: আপনার CI runners যদি Docker-in-Docker ব্যবহার করে, Podman-এ সমতুল্য কনফিগারেশন বিভিন্ন হতে পারে—পাইপলাইনের environment টেস্ট করুন।
5. Windows/macOS ডেভেলপমেন্ট: টিমে যদি বেশিরভাগ Windows/Mac ব্যবহার করে, Podman Machine বা Podman Desktop কেমন কাজ করে তা আগে যাচাই করুন।

টেকনিক্যাল রিকমেন্ডেশন: ছোট ধাপে মাইগ্রেট করুন—প্রথমে non-critical সার্ভিস নিয়ে পরীক্ষা করা, CI এ একটি স্টেজ চালিয়ে দেখা, তারপর প্রোড-স্টেজ।

## সাধারণ ভুল ধারণা ও সতর্কতা

- "Podman সব সমস্যার সমাধান" — না; এটা নির্দিষ্ট প্রেক্ষাপটে সুবিধা দেয়, কিন্তু পরিবেশভেদে কনফিগারেশন কাজ থাকতে পারে।
- "Docker image সবসময় একইভাবে চলবে" — অধিকাংশই ঠিক, কিন্তু ফাইল-পারমিশন বা volume মাউন্ট ক্ষেত্রে টূল-ভিত্তিক পার্থক্য থাকতে পারে।

## প্র্যাকটিক্যাল এক্সারসাইজ (হাতেকলমে যাচাই)

উদ্দেশ্য: CLI মিল/পার্থক্য দেখুন এবং ছোট-খাটো টেস্ট করে সিদ্ধান্ত গ্রহণ সহজ করুন।

প্রশিক্ষণ ধাপ (আপনি লিনাক্স/ম্যাক/উইন্ডোজ WSL এ কাজ করছেন ধরে নেওয়া):

1) কেবল পর্যবেক্ষণ: একই ইমেজ দু’টি টুল দিয়ে টানুন

- docker pull alpine:latest
- podman pull alpine:latest

লক্ষ্য: উভয়ের আউটপুট, সময় ও লেয়ার তথ্য লক্ষ্য করুন — যদি আপনার সিস্টেমে ডকার না থাকে, কেবল podman টেস্ট করুন।

2) ছোট রান টেস্ট

- docker run --rm alpine echo "hello from docker"
- podman run --rm alpine echo "hello from podman"

লক্ষ্য: কোনো পারমিশন ইস্যু, নেটওয়ার্ক সংক্রান্ত ভিন্নতা আছে কি না দেখুন।

3) প্রসেস তালিকা ও পদ্ধতি দেখুন

- docker ps -a
- podman ps -a

লক্ষ্য: পডম্যানের প্রক্রিয়াগুলো কীভাবে হোস্টে প্রতিফলিত হয় পরীক্ষা করুন (daemon vs per-command processes)।

4) (ঐচ্ছিক, উন্নত) systemd unit উৎপন্ন করা (লিনাক্সে)

- podman create --name hello-nginx nginx:alpine
- podman generate systemd --new --name hello-nginx

লক্ষ্য: generated unit ফাইল দেখুন—পডম্যান systemd ইন্টিগ্রেশনের সুবিধা কীভাবে কাজ করে তা দেখাবে।

5) রিফ্লেকশন লিখুন

- ৫-১০ লাইনে লিখুন: আপনি কোন পরিস্থিতিতে Podman বেছে নেবেন, আর কখন Docker রাখা বাঞ্ছনীয় মনে হবে।

নোট: যদি আপনার কাজের পরিবেশে কোনো নির্দিষ্ট টুলিং বা সার্ভিস (যেমন Docker Desktop, একটি CI runner) নির্ভর করে, সেগুলোকে পরীক্ষায় অন্তর্ভুক্ত করুন।

## সারমর্ম

- Podman ও Docker উভয়ই শক্তিশালী টুল; মিল অনেক, কিন্তু আর্কিটেকচারের গুরুত্বপূর্ণ পার্থক্য (daemon vs daemon-less, rootless) ব্যবহার শতর্ক করে।
- ছোট প্রকল্পে দ্রুত শুরু করার জন্য CLI মিল সুবিধা দেয়; কিন্তু বড় প্রোডাকশনে নিরাপত্তা এবং অপারেশনাল চাহিদা ভিত্তিক সিদ্ধান্ত নিন।

পরবর্তী লেসনগুলোতে আমরা ইনস্টলেশন, হ্যালো-ওয়ার্ল্ড রান এবং পডম্যান আর্কিটেকচারের গভীর ব্যাখ্যা দেব—সেখানে হাতে-কলমে কিভাবে পডম্যান সেটআপ করবেন দেখানো হবে।
