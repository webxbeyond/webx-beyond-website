## স্টেটলেস, ইউনিফর্ম ইন্টারফেস, রিসোর্স ডিজাইন

REST (Representational State Transfer) হলো ওয়েব API ডিজাইনের সবচেয়ে জনপ্রিয় স্টাইল। RESTful API সহজ, স্কেলেবল ও স্ট্যান্ডার্ডাইজড।

## ১. স্টেটলেস (Stateless)
- **সংজ্ঞা:** প্রতিটি রিকোয়েস্ট সম্পূর্ণ স্বতন্ত্র; সার্ভার ক্লায়েন্টের কোনো আগের অবস্থা (state) মনে রাখে না।
- **উদাহরণ:**
  - প্রতিবার API-তে রিকোয়েস্ট পাঠানোর সময় অথেন্টিকেশন টোকেন পাঠাতে হয়।
- **সুবিধা:** স্কেলিং সহজ, সার্ভার কমপ্লেক্সিটি কম।

## ২. ইউনিফর্ম ইন্টারফেস (Uniform Interface)
- **সংজ্ঞা:** সব রিসোর্সে একই নিয়ম/স্টাইল ফলো করা।
- **উদাহরণ:**
  - সব API-তে GET, POST, PUT, DELETE মেথড ব্যবহার।
  - রিসোর্স URL-এর গঠন একরকম (যেমন: `/users/123`, `/products/456`)
- **সুবিধা:** ডেভেলপারদের জন্য শেখা ও ব্যবহার সহজ।

## ৩. রিসোর্স ভিত্তিক ডিজাইন (Resource-Based Design)
- **সংজ্ঞা:** API-তে প্রতিটি ডেটা/অবজেক্টকে "রিসোর্স" হিসেবে ধরা হয়।
- **উদাহরণ:**
  - `/users` (সব ইউজার), `/users/123` (একজন ইউজার)
  - `/orders/789` (একটি অর্ডার)
- **সুবিধা:** স্পষ্ট, লজিক্যাল, এবং RESTful স্ট্যান্ডার্ডের সাথে সামঞ্জস্যপূর্ণ।

## REST-এর আরও কিছু মূল কনসেপ্ট
- **Client-Server Architecture:** ক্লায়েন্ট ও সার্ভার আলাদা, স্বাধীনভাবে ডেভেলপ/ডিপ্লয় করা যায়।
- **Cacheable:** রেসপন্স ক্যাশ করা যায়, ফলে পারফরম্যান্স বাড়ে।
- **Layered System:** একাধিক লেয়ার (প্রক্সি, গেটওয়ে) ব্যবহার করা যায়।

## উদাহরণ: একটি RESTful API
```
GET /api/products/123
```
- **Method:** GET
- **Resource:** products/123
- **Stateless:** প্রতিবার অথেন্টিকেশন টোকেন পাঠাতে হবে

## সংক্ষেপে
- REST API ডিজাইনের মূল ভিত্তি: স্টেটলেস, ইউনিফর্ম ইন্টারফেস, রিসোর্স ভিত্তিক ডিজাইন
- এগুলো মানলে API হবে সহজ, স্কেলেবল ও স্ট্যান্ডার্ড

> পরবর্তী অধ্যায়ে: RESTful এন্ডপয়েন্ট ডিজাইন—নাউন-ভিত্তিক URL, CRUD, ভার্সনিং
