## আইডেমপোটেন্সি, এরর হ্যান্ডলিং, HATEOAS

RESTful API ডিজাইনে কিছু বেস্ট প্র্যাকটিস মানলে API হবে আরও নির্ভরযোগ্য, ইউজার-ফ্রেন্ডলি ও স্কেলেবল।

## ১. আইডেমপোটেন্সি (Idempotency)
- **সংজ্ঞা:** একই রিকোয়েস্ট বারবার পাঠালেও একই রেজাল্ট আসবে।
- **উদাহরণ:**
  - `GET /users/123` বারবার করলে একই ডেটা পাবেন।
  - `DELETE /users/123` বারবার করলে ইউজার একবারই ডিলিট হবে, পরেরবার নো-অপারেশন।
- **কেন দরকার:** নেটওয়ার্ক সমস্যা বা ক্লায়েন্ট রিট্রাই করলে ডেটা ডুপ্লিকেট বা ভুল না হয়।

## ২. এরর হ্যান্ডলিং (Error Handling)
- **বেস্ট প্র্যাকটিস:**
  - স্পষ্ট ও কনসিস্টেন্ট এরর মেসেজ
  - HTTP স্ট্যাটাস কোড ব্যবহার (৪০০, ৪০৪, ৫০০ ইত্যাদি)
  - রেসপন্সে ডিটেইলড এরর অবজেক্ট
- **উদাহরণ:**
    ```json
    {
      "error": {
        "code": 400,
        "message": "Invalid email address"
      }
    }
    ```

## ৩. HATEOAS (Hypermedia as the Engine of Application State)
- **সংজ্ঞা:** রেসপন্সে পরবর্তী অ্যাকশন/লিঙ্ক দেয়া (নেভিগেশন সহজ করতে)।
- **উদাহরণ:**
    ```json
    {
      "id": 123,
      "name": "Rahim",
      "links": [
        { "rel": "self", "href": "/users/123" },
        { "rel": "orders", "href": "/users/123/orders" }
      ]
    }
    ```
- **সুবিধা:** ক্লায়েন্ট সহজে পরবর্তী রিসোর্স/অ্যাকশন খুঁজে পায়।

## সংক্ষেপে
- আইডেমপোটেন্সি API-কে নির্ভরযোগ্য করে তোলে
- এরর হ্যান্ডলিং ইউজার ও ডেভেলপার এক্সপেরিয়েন্স বাড়ায়
- HATEOAS API-কে আরও ডিসকভারেবল ও RESTful করে তোলে

> পরবর্তী অধ্যায়ে: বিকল্প এপিআই আর্কিটেকচার—GraphQL, gRPC, WebSocket, SOAP
