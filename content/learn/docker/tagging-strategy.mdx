## মূল ধারণা

### ১. Immutable Image: কী ও কেন?

- **Immutable image:** Once built/pushed—change করা যায় না।
- **Benefit:**
  - Predictable deploy
  - Rollback possible
  - Audit trail
- **Example:**
  - `myapp:1.0.0`—same image/version everywhere

### ২. Tagging Strategy Explained

- **Tag:** Image-এর unique identifier—version, environment, build info
- **Best practice:**
  - Semantic versioning: `myapp:1.0.0`, `myapp:2.1.3`
  - Environment tag: `myapp:prod`, `myapp:dev`
  - Build info: `myapp:20250826-commit1234`
- **Example:**
  ```bash
  docker build -t myapp:1.0.0 .
  docker push myapp:1.0.0
  docker run myapp:1.0.0
  ```

### ৩. Workflow: Versioned Deploy & Rollback

1. **Image build—specific tag/version**
2. **Image push—registry-তে version/tag সহ**
3. **Deploy—specific tag/version pull/run**
4. **Rollback—previous tag/version deploy**

### ৪. সহজ এনালজি

- "Immutable image মানে—একটি সিল করা বাক্স, যেটা খুললে আর পরিবর্তন করা যায় না। Tagging—প্রতিটি বাক্সে label, version, date লিখে রাখা।"

### ৫. Best Practices

- **Semantic versioning:** Major.minor.patch—predictable version
- **Automated tagging:** CI/CD pipeline-এ build info/date/tag add করুন
- **Immutable image enforced:** Once pushed—change না করুন
- **Audit trail:** Tag/version-based deploy/rollback
- **Environment-specific tag:** Prod/dev/test—clear separation

### ৬. সাধারণ ভুলত্রুটি

- Latest tag—production-এ use করা—unexpected change risk
- Manual tag/version—confusion, rollback issue
- Immutable image না enforced—unexpected change
- Audit trail/logging না রাখা—deploy history নেই

### ৭. Industry Trends

- **Automated tagging:** Git commit/date-based tag—CI/CD pipeline-এ
- **Immutable registry:** Image overwrite block—security enforced
- **Rollback automation:** Tag/version-based deploy—fast recovery

### ৮. কখন ব্যবহার করবেন?

- Production deploy—version/tag-based image essential
- CI/CD pipeline—automated tagging, rollback
- Audit trail—deploy history/version control
